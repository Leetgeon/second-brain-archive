# 프로젝트 계획

## 1. 목표

플레이리스트와 북마크에 흩어진 좋은 콘텐츠를 장기 보존하고, AI가 저장된 자료만을
근거로 검색·요약·비교·질의응답할 수 있는 개인 지식 아카이브를 만든다.

이 프로젝트의 성공 기준은 다음과 같다.

- URL 하나 또는 플레이리스트를 수집함에 넣을 수 있다.
- 원본을 보관할 권리가 있는 경우 미디어와 자막을 로컬에 보존할 수 있다.
- 모든 자료에 출처, 수집 시각, 권리 상태, 주제, 처리 상태가 남는다.
- AI 답변은 원본 URL과 영상 타임스탬프를 인용한다.
- 특정 AI 서비스가 사라져도 원본, 자막, 메타데이터는 계속 사용할 수 있다.
- 외장 디스크나 NAS로 이동하고 복구할 수 있다.

## 2. 제품 원칙

### 로컬 원본이 기준

클라우드 AI 서비스는 선택적인 소비자다. 아카이브의 기준 데이터는 로컬 파일,
SQLite, JSON, Markdown처럼 이식 가능한 형식으로 둔다.

### 원본과 파생 데이터 분리

- 원본: 미디어, 제공받은 자막, 원문 파일
- 정규화 데이터: 전사문, 챕터, 메타데이터, 태그
- 파생 데이터: 요약, 청크, 임베딩, AI 노트

파생 데이터는 언제든 삭제하고 다시 만들 수 있어야 한다.

### 권리 상태를 데이터로 관리

각 항목은 `owned`, `licensed`, `public-domain`, `reference`, `unknown` 중 하나의
권리 상태를 가진다. 자동 미디어 수집은 허용 상태에서만 실행하도록 설계한다.

### AI 답변에는 근거가 있어야 함

요약과 답변에는 자료 ID, 원본 URL, 자막 구간, 영상 타임스탬프를 함께 반환한다.
사용자가 원문으로 즉시 돌아갈 수 없는 답변은 완성된 답변으로 취급하지 않는다.

## 3. 사용자 흐름

1. 사용자가 URL, 플레이리스트 또는 로컬 파일을 수집함에 넣는다.
2. 시스템이 중복 여부와 권리 상태를 확인한다.
3. URL 자료는 메타데이터를 보관하고, 허용된 자료만 원본을 가져온다.
4. 제공 자막이 있으면 보존하고, 없으면 권한 있는 미디어를 로컬 음성 인식한다.
5. 전사문을 타임스탬프 단위로 정리하고 주제·인물·키워드를 추출한다.
6. 텍스트 검색과 벡터 검색 인덱스를 만든다.
7. 사용자는 질문, 비교, 요약, 복습 카드 생성을 요청한다.
8. 결과는 출처 링크와 타임스탬프를 포함해 Obsidian 또는 Markdown으로 내보낸다.

## 4. 시스템 구조

```text
[Capture]
 URL / Playlist / Local File / Platform Export
                 |
                 v
[Catalog]
 SQLite + JSON manifests + rights status
                 |
                 v
[Archive]
 media / transcripts / documents / thumbnails
                 |
                 v
[Processing]
 transcription -> normalization -> chapters -> tags -> chunks
                 |
                 v
[Retrieval]
 SQLite FTS5 + optional vector index
                 |
                 v
[Use]
 search / grounded Q&A / summaries / Obsidian export
```

## 5. 단계별 로드맵

### M0. 보관 기반

상태: 완료

- 로컬 아카이브 초기화
- URL 및 로컬 파일 등록
- SHA-256 중복 제거
- 권리 상태와 출처 기록
- SQLite와 JSON 매니페스트 생성

완료 조건: 샘플 URL과 로컬 파일을 등록하고 재조회할 수 있다.

### M1. URL과 플레이리스트 수집

상태: 완료

- 웹 대시보드와 CLI에서 URL 추가
- 공식 YouTube OAuth로 내 계정 재생목록 조회
- 재생목록 항목의 일괄 카탈로그 등록 및 선택적 원본 수집
- yt-dlp를 통한 플레이리스트 항목·메타데이터 동기화
- 재실행 가능한 동기화 로그
- `personal-use`를 포함한 권리 상태 기록

완료 조건: 플레이리스트 변경분만 반복 수집하며 중복 항목이 생기지 않는다.

### M2. 자막과 전사

상태: 완료

- 사용자가 제공한 SRT/VTT 보존
- yt-dlp가 제공하는 원본·자동 자막 보존
- 미디어의 로컬 whisper.cpp 전사
- 타임스탬프가 있는 검색 청크와 Markdown 생성

완료 조건: 영상의 특정 문장을 검색해 해당 시점으로 이동할 수 있다.

### M3. 검색과 AI 질의응답

상태: 완료

- SQLite FTS5 키워드 검색
- 90초 이하의 근거 청크 규격
- 한국어 조사를 고려한 FTS5와 부분 문자열 검색
- 일반 질문을 위한 최근 근거 구간 보완
- 답변마다 자료명, URL, 타임스탬프 인용

완료 조건: 저장 자료에 없는 내용은 모른다고 답하고, 있는 내용은 근거를 제시한다.

### M4. 사용 화면과 Obsidian 연동

상태: 완료

- 수집함, 처리 대기열, 검색, 자료 상세 화면
- Obsidian용 Markdown 노트 내보내기
- 로컬 Ollama를 통한 근거 기반 답변 생성

완료 조건: 터미널 없이 수집부터 검색·노트 생성까지 수행할 수 있다.

### M5. 운영과 백업

상태: 기본 기능 완료

- 원본 SHA-256 체크섬과 중복 제거
- 3-2-1 백업 지침
- `--root`와 `SECOND_BRAIN_HOME`을 통한 외장 디스크/NAS 경로 지원
- SQLite와 원본 SHA-256 무결성 검사
- 모델 이름을 답변과 내보내기 결과에 기록

완료 조건: 새 컴퓨터에서 백업만으로 카탈로그를 복구할 수 있다.

## 6. 우선 기술 선택

- 언어: Python 3.11+
- 카탈로그: SQLite
- 텍스트 검색: SQLite FTS5
- 원본 저장: 파일 시스템 + SHA-256 주소
- 전사: 로컬 Whisper 계열 어댑터
- AI 검색: Ollama 로컬 API
- UI: Python 표준 라이브러리 기반 경량 웹 앱
- 노트 교환: Markdown + YAML front matter

벡터 데이터베이스는 자료 규모가 커질 때 도입한다. 초기에는 SQLite FTS5와 작은
임베딩 인덱스로 충분한지 먼저 측정한다.

## 7. 데이터 모델

각 자료는 최소한 다음 필드를 가진다.

| 필드 | 의미 |
|---|---|
| `id` | 내부 고유 ID |
| `title` | 자료 제목 |
| `kind` | `url` 또는 `file` |
| `source_url` | 원본 출처 |
| `platform` | YouTube, podcast, web 등 |
| `playlist` | 사용자가 분류한 컬렉션 |
| `rights_status` | 소유·이용 권리 상태 |
| `rights_note` | 허용 근거와 제한 |
| `media_sha256` | 원본 파일 체크섬 |
| `media_path` | 아카이브 내 원본 위치 |
| `transcript_path` | 자막 또는 전사문 위치 |
| `created_at` | 등록 시각 |

구현된 추가 필드는 `published_at`, `creator`, `duration_seconds`,
`thumbnail_path`, `description_path`, `status`, `updated_at`이다.

## 8. 첫 의사결정

초기 제품은 YouTube에 한정되지 않는다. yt-dlp가 지원하는 여러 플랫폼의 자료를
같은 규격으로 보존한다.

첫 번째 실제 사용 시나리오는 다음으로 고정한다.

1. 영상 또는 플레이리스트 URL을 `personal-use` 상태로 수집
2. 원본 자막을 보존하거나 로컬 Whisper 전사
3. 자막 전체 검색과 원본 타임스탬프 이동
4. Ollama에 근거 기반 질문
5. 영상 노트 또는 답변을 Markdown으로 내보내기
